03
Memory Modules · 四类记忆
短期工作记忆 + 三类长期记忆
MEMORY

CoALA 明确区分“当前正在处理的状态”和“可跨任务复用的长期资产”。这个区分决定了 Agent 是临时拼 prompt，还是能形成可迭代的系统。

01 · Short-Term
工作记忆 Working Memory
当前决策循环里可直接使用的变量，包括观察、目标、已取回知识、推理中间结果和待执行动作。它不是 LLM 上下文的同义词，而是跨多次模型调用持续存在的数据结构。
  • 负责把环境输入、检索结果和模型输出组织成可执行状态。
  • 每次 LLM 调用只读取其中一部分，再把结构化结果写回。
02 · Experience
情景记忆 Episodic Memory
保存过去决策循环和任务轨迹，例如事件流、输入输出样例、游戏轨迹、失败案例和用户交互历史。规划时可取回相似经历，作为示例或反思材料。
  • 写入对象是“发生过什么”。
  • 价值在于复盘、类比、少样本示例和行为连续性。
03 · Knowledge
语义记忆 Semantic Memory
保存关于世界和智能体自身的知识。它可以来自外部语料，也可以由 Agent 从经验中反思、总结和写入。
  • 写入对象是“这意味着什么”。
  • 不应只做只读 RAG，应支持事实沉淀、来源、置信度和失效管理。
04 · Procedure
程序记忆 Procedural Memory
保存“怎么做”的知识，包括 LLM 参数中的隐式能力，以及 Agent 代码中的显式流程、工具封装、提示模板、解析器和可复用技能。
  • 写入对象是“以后如何行动”。
  • 修改程序记忆最强，也最危险，必须有测试、权限和回滚。
结构性判断
记忆类型 主要载体 典型读操作 典型写操作
工作记忆 结构化变量、当前上下文 构造 prompt、选择动作 解析模型输出、更新目标和观察
情景记忆 事件日志、任务轨迹 相似案例、失败复盘 记录新经历、保存轨迹片段
语义记忆 知识库、文档、事实表 检索知识、补全背景 从经验中总结事实和偏好
程序记忆 代码、技能库、提示模板、模型参数 加载工具、调用技能、选择策略 新增技能、改写提示、微调模型

04
Actions & Loop · 读写机制
记忆通过内部动作进入决策循环
LOOP

CoALA 的关键点是：读取记忆、推理、写入记忆都不是幕后副作用，而是 Agent 动作空间的一部分。这样才能讨论权限、成本、风险和可测试性。

Read · Long-Term → Working
Retrieval 取回
从长期记忆中读取信息到工作记忆。实现方式可以是规则、关键词、向量检索、代码技能检索，或多信号排序。
recency importance relevance
Transform · Working → Working
Reasoning 推理
读取工作记忆，再产生新的工作记忆变量。例如总结观察、反思失败、生成候选动作、评估计划和更新目标。
summarize reflect evaluate
Write · Working → Long-Term
Learning 学习
把经验、知识或程序写入长期记忆。论文特别强调：学习不只是微调，也包括存储经历、总结知识和新增代码技能。
episode fact skill
External · Agent → World
Grounding 接地
与外部环境交互，并把反馈转成文本观察写入工作记忆。环境可以是物理世界、人类对话、网站、API、代码执行或游戏。

决策循环 · Planning and Execution
01
Proposal
基于工作记忆、取回内容和推理，生成一个或多个候选动作。
02
Evaluation
用规则、价值模型、LLM 推理或模拟，对候选动作估值。
03
Selection
从候选动作中选择一个，或拒绝并回到生成阶段。
04
Execution
执行外部接地动作或内部学习动作，观察反馈后进入下一轮。

工程启发：每个记忆动作都应该像工具调用一样有输入、输出、权限、成本、日志和验收条件。否则“记忆”会退化成不可解释的副作用。

05
Engineering Blueprint · 落地模型
从论文框架翻译成系统设计
BUILD

把 CoALA 落地为产品或开发者框架时，不应先问“用哪种向量库”，而应先定义记忆对象、生命周期和读写策略。

Layer 1
Working State
当前任务的结构化状态：目标、约束、观察、已检索材料、候选动作、评估结果、待确认事项。它是每轮决策的输入输出总线。
Layer 2
Episodic Log
不可变事件流：任务、用户反馈、环境观察、工具调用、失败原因、最终结果。先保留证据，再决定是否总结成知识。
Layer 3
Semantic Store
可检索知识：事实、偏好、规则、结论和约束。每条知识要有来源、时间、置信度、适用范围和失效条件。
Layer 4
Procedure Registry
技能与流程：工具封装、提示模板、解析器、工作流和代码技能。变更需要版本化、测试、人工审核和回滚机制。
一个可复用的记忆管线
阶段 要回答的问题 关键设计
Capture 什么值得记录 区分原始事件、用户显式反馈、系统推断；保留来源和时间。
Consolidate 如何从经历变成知识 用反思或规则抽取事实，但保留证据链，避免单次失败过度泛化。
Retrieve 何时取回什么 同时考虑相关性、近期性、重要性、任务阶段和记忆类型。
Apply 如何进入决策 把记忆写入工作状态，而不是直接拼进一大段 prompt。
Govern 如何修正和遗忘 支持冲突检测、人工编辑、过期、删除、版本和权限边界。

好的 Agent 记忆系统不是“召回更多内容”，而是让正确的记忆在正确的时刻以正确的结构进入工作状态。

06
Design Rules · 设计原则
用 CoALA 检查一个 Agent 的记忆能力
RULES

如果要评估一个 Agent 是否真的具备可成长的记忆能力，可以直接用下面的结构性清单，而不是看它是否接入了某个存储组件。

Rule 01
先分清记忆类型
当前变量、历史经历、世界知识和可执行技能不能混在同一个“memory”字段里。混在一起会导致检索污染和权限失控。
Rule 02
把读写动作显式化
检索、推理、学习都应是可追踪动作。每次读写都应该知道触发条件、输入、输出、成本和失败模式。
Rule 03
让工作记忆成为中枢
长期记忆不要直接塞进模型上下文，而要先进入结构化工作状态，再由决策过程选择性使用。
Rule 04
程序记忆必须更严格
让 Agent 改提示、改技能、改代码是强能力，但会引入功能和对齐风险。需要版本、测试、审批和回滚。
常见风险与修正
RAG ≠ Memory
只读外部文档库只是语义记忆的一种读取方式。Agent 记忆还包括情景记忆、工作记忆、程序记忆以及写入和治理。
Recall ≠ Use
召回内容不代表会正确使用。必须通过工作记忆和决策循环，让记忆参与候选动作生成、评估和选择。
Write ≠ Learn
把所有对话都存下来不是学习。学习需要筛选、抽象、验证、冲突处理和可撤销的更新机制。
Code = Risk
写入程序记忆会改变 Agent 的行为边界。越接近代码、工具和决策流程，越需要安全策略和人工控制。

最终结论：智能体记忆的核心不是存储，而是一个闭环：经历被记录，知识被抽象，相关内容被取回，工作状态被更新，行动被改变，再产生新的经历。
