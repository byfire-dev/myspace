Vibe Coding Principle

ARCHITECTURE
FIRST

结构性问题，架构性解决

对结构性问题，优先做架构性解决，而不是 case-by-case 补丁。先抽象底层模型，设计完整方案，再实现可复用模块，并用关键场景验证，确保一次性覆盖同类问题。

CORE RULE

当问题反复出现、分支不断增加、修复依赖人肉判断时，不要继续补洞。先把问题抽象成模型，再让实现服从模型。

OUTPUT SHAPE

底层模型 → 完整方案 → 可复用模块 → 关键场景验证 → 同类问题一次覆盖。

Architecture Model Reusable Validation
SMELL
识别结构性问题，而不是把它当成单点 bug

如果同一种异常在不同页面、不同状态、不同数据源里轮流冒出来，真正的问题通常不在某一行代码，而在缺少统一抽象、边界或流程。

  • 同类 if/else、兜底逻辑、格式转换散落多处。
  • 每修一个 case，又出现两个相邻 case。
  • 修复依赖“记得这样写”，而不是模块自动保证。
SHIFT
从补丁思维切到模型思维

case-by-case 的问题是局部正确、全局脆弱；架构性解决的目标是让正确路径变成默认路径，让新场景天然落入同一个处理框架。

PATCH MODE遇到一个 case，补一个分支；需求越多，系统越靠记忆维持。
→
MODEL MODE先定义对象、状态、规则和边界；同类问题统一走模块。
DECISION
什么时候值得上架构解

不是所有 bug 都要抽象。判断标准是复用面、变化频率和风险半径：只要后续会不断遇到同族问题，就该把解法做成系统能力。

  • 同类逻辑会出现在三个以上入口。
  • 业务规则还会继续增长或改口径。
  • 错误会影响数据一致性、权限、交易或用户信任。
01
抽象底层模型

把表象 case 收拢成对象、状态、约束、输入输出和不变量。先回答“这类问题到底是什么”。

02
设计完整方案

定义模块边界、数据流、失败路径、迁移方式和兼容策略。方案要覆盖新增、修改、异常和回滚。

03
实现可复用模块

把分散逻辑沉到统一函数、组件、服务或策略表里，让调用方少选择、少复制、少发挥。

04
验证关键场景

用代表性场景覆盖主路径、边界、历史数据、并发或权限差异，证明模型真的吃得下同类问题。

05
沉淀使用规则

把接入方式、禁止写法、测试样例和迁移检查写进文档、AGENTS.md 或 Skill，避免退回补丁模式。

MODEL MAP
底层模型至少回答四个问题

抽象不是起名字，而是找到稳定的结构。一个好模型能解释旧 case，也能约束新 case。

对象系统里真正被处理的实体是什么？它有哪些必要字段和生命周期？
状态它会在哪些状态间切换？哪些状态是非法或临时的？
规则哪些判断必须集中管理？哪些分支不能散落在调用方？
边界模块对外暴露什么，不暴露什么？谁负责兜底和报错？
变化未来最可能新增的是类型、来源、步骤还是展示方式？
证据哪些测试或数据样例能证明模型不是只适配当前 case？
VALIDATION MATRIX
不要只验证刚刚坏掉的那个点

关键场景验证要覆盖同类问题的代表性切片，避免“当前 case 绿了，邻居 case 又红”。

主路径最常见输入、正常状态、标准输出。
边界值空值、未知类型、极大极小、缺字段。
历史包袱旧数据、旧接口、旧配置仍能安全接入。
新增入口新页面、新渠道、新调用方不用复制逻辑。
失败路径错误可预期、可恢复、可观测。
回归保护测试锁住模型规则，而不是锁住偶然实现。
ARCHITECTURE PROMPT
先让 Codex 建模，不要直接修 case

当问题疑似结构性时，第一轮 prompt 要阻止它马上写补丁。

下面是一个反复出现的问题：
<描述现象、相关 case、当前补丁>

请先不要写代码。请判断它是不是结构性问题，并输出：
1. 同类问题的底层模型
2. 当前 case 只是哪个表象
3. 应该集中到哪个模块或边界
4. 完整方案和迁移步骤
5. 关键验证场景
6. 哪些 case-by-case 修复应该删除或收敛
IMPLEMENTATION PROMPT
实现时要求模块化和场景证明

第二轮再写代码，并明确要求可复用边界和验证证据。

请按上面的架构方案实现。

约束：
- 不新增散落的 case-by-case 分支
- 把规则集中到可复用模块
- 保持调用方 API 简单
- 补齐关键场景测试
- 删除被新模块替代的重复逻辑

完成时请说明：
1. 新模型/模块的边界
2. 覆盖了哪些同类场景
3. 验证命令和结果
4. 后续新增 case 应该如何接入
NORTH STAR
让正确做法
成为最省力的做法

架构性解决不是为了显得高级，而是为了减少未来每一次判断、复制和补洞。好的模块会让调用方自然走进正确路径。
