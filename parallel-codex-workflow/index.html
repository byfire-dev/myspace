VIBE CODING OPERATING MODEL

PARALLEL
CODEX WORKFLOW

把 Vibe Coding 从对话升级为多 Agent 调度系统

瓶颈不在于单个 Codex 回答不够快，而在于工作流仍停留在“一个人、一个聊天框、一个任务串行等待”。更高效的模式，是把需求拆成任务卡，让多个线程或 worktree 并行推进，用自动验收和 review agent 控制质量，人类只在调度、取舍和合并处介入。

CORE SHIFT

从“逐句提示 AI”转向“管理一组 coding agents”：你不再负责每一步怎么写，而是负责定义任务边界、并行策略、验收标准和最终合并顺序。

SYSTEM SHAPE

任务拆分 → 多线程 / 多 worktree 并行 → 自动验收 → 实现 agent + review agent + 集成 agent → 人类合并。

BOTTLENECK
不是 Codex 慢，而是你在串行排队

旧模式是“任务 A → 等 Codex → review → 修正 → 任务 B”。只要你仍把 Codex 当成单个聊天对象，吞吐量就被你的等待、上下文复述和临时验收卡住。

CHAT MODE一个问题、一个线程、一个结果，所有任务共享同一个等待队列。
→
OPS MODE多张任务卡并行流动，人类只处理阻塞、review 和合并。
TASK CARD
把“问问题”改成“发任务卡”

任务卡要让 agent 少猜：说清目标、上下文、约束和完成标准。OpenAI 的 Codex 最佳实践也建议 prompt 包含 Goal、Context、Constraints、Done when 这四类信息。

Goal
实现用户设置页头像上传，支持 png/jpg。
Context
相关目录、参考组件、接口文档和现有错误处理方式。
Limits
不改 schema，不加依赖，只改功能相关文件。
Done
失败提示、测试补充、验证命令和改动摘要。
CONTROL POINT
人类的工作变成调度，而不是陪跑

你应该把注意力放在任务是否可并行、谁先谁后、哪些文件会冲突、哪些结果值得合并，而不是重复解释“不要用 any”“先跑测试”“别动 unrelated files”。

Scope Review Merge Rules
01
拆任务

先让 Codex 产出可并行子任务：目标、涉及文件、冲突风险、建议 worktree 名称和完成标准。

02
开轨道

修 bug、补测试、写文档、做方案探索，可以分配到不同 thread 或 Git worktree 中并排推进。

03
自动验收

每张卡都带验证命令。lint、typecheck、单测、截图或复现脚本，是 agent 完成任务前必须交付的证据。

04
交叉 review

实现 agent 只负责完成需求；review agent 只看 diff、边界条件、测试缺口和不必要改动。

05
集中集成

通过 review 的任务进入集成队列。人类最终决定合并顺序、冲突处理和是否拆小 PR。

PARALLEL LANES
适合并行

独立文件、只读分析和验证型任务最适合并行，收益最大、冲突最少。

  • 补单元测试、修不同页面 bug、写文档
  • API 客户端封装、迁移计划、性能分析
  • lint / type error 修复、PR 安全与测试 review
CAREFUL LANES
谨慎并行

会碰核心文件或需要统一产品判断的任务，先拆计划，再决定是否分轨。

  • 同一个核心模块、数据库 schema、认证权限
  • 支付、跨模块状态管理、大范围 UI/UX 判断
  • 全面重构前，先让 Codex 输出分阶段方案
WORKTREE RULE
用 worktree 隔离改动

Codex app 的 Worktree 模式会创建独立 Git worktree，让多个任务在同一项目中隔离工作，适合尝试新想法或并排处理独立任务。

  • worktree-1：修 bug A
  • worktree-2：实现功能 B
  • worktree-3：补测试 C
AGENT ROLES
不要让同一个线程写代码、评审、合并

复杂任务要拆成角色，而不是让一个 agent 自己证明自己写得好。实现、review、集成三者分离，会让问题更早暴露，也更容易把责任范围说清。

Agent A · 实现只关注完成需求、保持范围、补齐验证。
Agent B · Review只看 diff、边界、回归、测试缺口。
Agent C · 集成根据 review 修复、整理最终 PR、解释风险。
PERSISTENT CONTEXT
把重复要求写进 AGENTS.md 和 Skills

如果你反复提醒“不要新加依赖”“测试命令是 pnpm test”“PR 摘要要写风险”，说明这些规则应该固化。AGENTS.md 适合项目规则，Skills 适合可重复流程。

AGENTS.md编码规范、验证命令、PR 输出格式、项目边界。
$pr-review安全、测试、回归风险、可维护性检查。
$bug-fix复现、定位、最小修复、补测试、验证命令。
$test-writer只补测试，不改业务逻辑，覆盖关键路径。
SPLIT PROMPT
先拆并行计划，不要立刻写代码

当任务大、风险高、冲突文件不明时，先让 Codex 拆任务和排集成顺序。

我想把下面的需求改成并行 Codex 工作流。

需求：
<粘贴需求>

请先不要写代码。请输出：
1. 任务拆分
2. 哪些任务可以并行
3. 哪些任务必须串行
4. 每个任务建议的 Codex prompt
5. 建议使用 Local / Worktree / Cloud
6. 完成标准
7. 最终集成顺序
8. 潜在冲突文件
REVIEW PROMPT
给 review agent 的指令要像检查清单

Review 线程不要改代码，只输出风险等级和给实现 agent 的精确修改指令。

你是 review agent。请只 review 当前分支相对 main 的 diff，不要修改代码。

重点检查：
- 是否满足需求
- 是否破坏现有行为
- 是否有竞态、空值、权限、错误处理问题
- 测试是否覆盖关键路径
- 是否有不必要的大范围改动

输出：
1. 必须修复
2. 建议修复
3. 可以接受
4. 给实现 agent 的精确修改指令
DAILY RHYTHM
早上发卡，白天调度，晚上合并

早上列出 5-10 个任务卡并标记并行性；白天只看进度、处理阻塞和调整方向；晚上集中 review diff、合并小 PR，并把重复问题写回 AGENTS.md 或 Skill。

FIRST THREE BETS

固化项目规则
02用 Worktree

pr-review / bug-fix

效率提升通常不是来自单次回答更快，而是来自同时有多个 agent 在推进，且你只在关键节点介入。
