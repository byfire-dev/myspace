Experience Note · Claude Code A5 / 01

CLAUDE
FABLE 5

从监督执行，转向管理方向、背景和验证

这份经验文档整理自一段 Claude Code 团队口播：当模型能长时间自主推进、写代码并测试成果时，人类的工作重心要改变。重点不再是盯着模型每一步是否正确，而是确认它是否在做“该做的事”。

核心转变：不要只把 Claude 当成执行工具，要把它当成会思考、会推进、会验证的协作者。

你提供足够背景、明确目标和验收方法；它负责探索路径、实现方案和自我检查。人的价值上移到方向判断、框架设计和边界把控。

原始内容强调三件事：把 Claude 当思考伙伴、给 Claude 目标和验证方法、让 Claude 尝试过去以为大模型做不到的事。

Mental Model

真正的问题变了：从“它做得对不对”到“它是不是在做该做的事”

A5 / 02
旧模型：逐步监督 把任务拆很小，反复检查输出，担心它中途停下或跑偏。人类像监工，模型像执行员。
→
新模型：方向管理 模型能连续推进、测试成果、提出更好实现。人类主要定义目标、背景、验证和取舍。
Principle 01

把 Claude 当成思考伙伴

在需求还模糊、你也不确定能不能做的时候，越应该早拉 Claude 进来。先让它采访你、拆问题、指出不确定性，而不是等到自己想清楚后再把任务丢给它执行。

  • 先写一份小 SPEC，让 Claude 反向提问，把隐藏需求挖出来。
  • 让 Claude 生成多种方向或 HTML 原型，用可见方案帮助你做判断。
  • 给背景而不只是约束，让模型理解为什么这样做。
Principle 02

给目标，也给验证方法

强模型需要的不只是“做完这个功能”，还需要知道什么叫完成。目标让它有空间发挥，验证方法让它不会偏离你真正要的结果。

  • 把 SPEC 作为目标来源，不让模型凭感觉定义范围。
  • 让它逐项验证计划，输出报告和差异说明。
  • 允许创造性实现，但用验收清单锁住结果。
Principle 03

更大胆地使用模型

如果你一直认为大语言模型做不到某件事，不妨把它变成一次实验。视频剪辑、复杂原型、跨文件重构、长时间自主 coding，都是值得重新测试的能力边界。

Operating Workflow

一套可复用的 Claude 协作流程

A5 / 03
01

先写最小 SPEC

只写目标、背景、已知约束和不确定问题。不要急着规定实现细节。

02

让 Claude 采访你

要求它反向提问，补齐边界、输入输出、用户场景、失败路径和验收标准。

03

让 Claude 重写最终 SPEC

把问答整理成明确目标、非目标、方案边界、实现计划和验证清单。

04

把 SPEC 变成 Goal

要求 Claude 持续推进到目标完成，中途遇到分歧时记录假设和取舍。

05

用 Workflow 验证成果

让 Claude 按验收清单逐项检查，生成完成报告：做了什么、怎么验证、哪里和原计划不同。

提示词里的关键变化：少说“别过度设计”，多说“为什么不要过度设计”。

例如：不要只写“保持简单”。更好的写法是：“这是一个实验功能，很可能一个月后删除，不要搭任何未来扔掉会心疼的东西。”背景越真实，模型越能做出符合目标的工程判断。

Reusable Prompts

可直接复用的提示词模板

A5 / 04
Prompt 01 · 澄清需求

让 Claude 先采访你，而不是直接开工

我有一个初步想法，但还没完全想清楚：
<粘贴想法或小 SPEC>

请先不要实现。请像产品/工程伙伴一样采访我：
1. 找出目标、用户、边界和风险里的模糊点
2. 按优先级列出你需要问我的问题
3. 说明哪些问题会影响架构或实现方向
4. 等我回答后，再帮我整理最终 SPEC
Prompt 02 · 目标与验证

让 Claude 自主推进，但必须交付证据

请把下面的 SPEC 作为目标完整实现：
<粘贴最终 SPEC>

执行要求：
- 允许你选择合适的实现路径
- 遇到不确定点先记录假设
- 不要只完成表面功能，要补齐关键验证

完成后请输出报告：
1. 实现了什么
2. 运行了哪些验证
3. 哪些地方与原计划不同，为什么
4. 还剩哪些风险或后续建议
Prompt 03 · 能力探索

把“不知道能不能做”改成一次实验

我过去以为大语言模型不擅长做这件事：
<描述任务>

请把它当成一次能力探索：
1. 先提出 3 种可行方向
2. 每种方向说明预期效果、成本和风险
3. 选一个最值得试的方向做最小原型
4. 给出判断是否值得继续投入的验收标准
