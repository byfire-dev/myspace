模型 Provider
Hermes Agent 把「记忆 + 技能 + 检索 + 用户建模」编织成一条持续运转的回路，让 agent 把一次性经验沉淀为可复用、可维护、可回收的长期资产——这是它相对传统「无状态 Agent 框架」的本质差异。下面是六大产品定位。

🔁闭合学习回路

agent 自管记忆 + 周期 nudge；复杂任务后自主造技能；技能使用中自我改进；FTS5 会话检索 + Honcho 辩证用户建模。

🧩模型无关

Nous Portal / OpenRouter(200+) / GLM / Kimi / MiniMax / OpenAI / Anthropic 原生 / 自建端点。hermes model 一键切换，无锁定。

💬无处不在

单一网关进程服务 Telegram、Discord、Slack、WhatsApp、Signal、Email、CLI…跨平台连续，支持语音转写。

☁️随处部署

local / Docker / SSH / Daytona / Singularity / Modal 六后端；Daytona+Modal serverless 持久化（空闲休眠、按需唤醒）。

🧪研究友好

批量轨迹生成、Atropos RL 环境、轨迹压缩，直接产出训练下一代工具调用模型的数据。

⚙️平台无关内核

同一个 AIAgent 服务 CLI/Gateway/ACP/Batch/API Server，平台差异只在入口层。

02

核心：Agent Loop（AIAgent）

run_agent.py 中的 AIAgent 是整套系统的同步编排引擎（15k+ 行），统一负责 prompt 组装、provider 选择、可中断模型调用、串/并行工具执行、压缩/重试/回退、跨父子代理迭代预算，以及上下文丢失前刷写记忆。

🚪 两个入口

chat(msg) —— 薄封装，返回最终回复字符串。

run_conversation(...) —— 完整接口，返回含 messages / metadata / usage 的 dict；省略 system/history 时自动构建与加载。

🔀 三种 API 模式

全部收敛到统一内部消息格式（OpenAI 风格），调用前后做转换。

chat_completionscodex_responsesanthropic_messages

解析序：显式 arg → provider 探测 → base URL 启发 → 默认 chat_completions。

▸单轮生命周期（9 步）turn lifecycle 
run_conversation()
  1. 生成 task_id（若无）
  2. 追加用户消息到历史
  3. 构建/复用已缓存系统 prompt（prompt_builder）
  4. 预检是否需压缩（>50% 上下文）
  5. 由历史构建 API 消息（按 3 种模式分别转换）
  6. 注入临时 prompt 层（预算告警 / 上下文压力）
  7. Anthropic 上应用 prompt 缓存标记
  8. 发起可中断 API 调用
  9. 解析响应：
      • 有 tool_calls → 执行 → 追加结果 → 回到 5
      • 纯文本 → 持久化 → 必要时刷写记忆 → 返回
消息交替规则：系统消息后必须 User↔Assistant 严格交替；工具阶段 Assistant(tool_calls)→Tool…→Assistant；绝不连续两条同为 user/assistant；只有 tool 角色可连续。Provider 会校验并拒绝畸形历史。
▸可中断 API 调用 + 工具执行interruptible · concurrency

可中断：_interruptible_api_call() 在后台线程发 HTTP，主线程同时等「响应就绪 / 中断事件 / 超时」。被中断（新消息、/stop、信号）时放弃 API 线程并丢弃响应，不注入任何半成品。

工具执行：单调用主线程直接执行；多调用经 ThreadPoolExecutor 并发，交互式工具（如 clarify）强制串行，结果按原始顺序回插。每个调用走「解析 handler → pre-hook → 危险命令检测/审批 → 执行 → post-hook → 追加 tool 结果」。

Agent 级工具（被循环提前拦截，直接改 agent 状态）：

todomemorysession_searchdelegate_task
▸回调面 · 迭代预算 · 回退callbacks · budget · fallback

回调面（平台无关关键）：tool_progress / thinking / reasoning / clarify / step / stream_delta / tool_gen / status，让同一内核在终端显 spinner、在聊天发进度、在编辑器更新状态。

迭代预算：IterationBudget 默认 90 轮（agent.max_turns）；子代理独立预算上限 50（delegation.max_iterations）；到 100% 停止并返回工作总结。

回退：主模型 429/5xx/401/403 时依次试 fallback_providers；_try_activate_fallback() 就地替换 model/provider/base_url/api_mode/client 并重评缓存，单次激活。辅助任务与子代理走各自独立链路；Cron 支持回退。

03

Prompt 组装与缓存稳定性

全项目最重要的设计决策之一：把可缓存的系统 prompt 状态与调用时临时附加项分离，直接影响 token 用量、缓存命中、会话连续性与记忆正确性。

可缓存系统 prompt 的 10 层（按序）

1 Agent 身份（SOUL.md → 否则默认） 2 工具感知行为指引 3 Honcho 静态块（启用时） 4 可选系统消息 5 冻结的 MEMORY 快照 6 冻结的 USER 画像快照 7 技能索引 skills index 8 上下文文件 AGENTS.md/.cursorrules… 9 时间戳 / 会话 ID 10 平台提示

调用时临时层（不进缓存 prefix）

ephemeral_system_promptprefill 消息网关会话上下文覆盖后续轮 Honcho recall

这些故意不持久化进系统 prompt，以保持稳定 prefix 不被打断 → 高缓存命中。

冻结快照语义：记忆/画像在会话开始注入，中途写入立即落盘但不改写已构建 prompt，到下一会话才生效；工具响应始终展示实时状态。专为保护 prefix 缓存。

SOUL.md 与上下文文件优先级（首个命中即用）

优先级文件搜索范围说明
1.hermes.md / HERMES.mdCWD → git 根Hermes 原生项目配置
2AGENTS.md仅 CWD（会话中渐进发现子目录）通用 agent 指令
3CLAUDE.md仅 CWDClaude Code 兼容
4.cursorrules / .cursor/rules/*.mdc仅 CWDCursor 兼容

所有上下文文件均经安全扫描（隐藏 unicode、"ignore previous instructions"、凭据外泄）、按 70/20 头尾比截断到 20,000 字符、剥离 YAML frontmatter。SOUL.md 作身份单独加载（skip_soul=True 防重复出现）。

04

上下文压缩与 Prompt 缓存

可插拔的 ContextEngine 抽象 + 双层压缩 + Anthropic prompt 缓存，共同管理长对话的上下文窗口。

图 2 · 双层压缩系统
▸压缩算法 4 阶段context_compressor.py

① 清理旧工具结果（廉价无 LLM）：保护尾部外 >200 字符的旧工具输出 → [Old tool output cleared…]。

② 确定边界：头部保护 [0..2]；中段摘要；尾部按 token 预算从后累加；_align_boundary_backward() 保证不切断 tool_call/result 组。

③ 生成结构化摘要：中段单次 call_llm(task="compression")，固定模板 Goal / Constraints / Progress(Done·In Progress·Blocked) / Key Decisions / Relevant Files / Next Steps / Critical Context。

④ 拼装：头部（首压追加 note）+ 摘要（角色避免同角色连续）+ 尾部原样；_sanitize_tool_pairs() 清理孤立 tool 对。

迭代再压缩：把上次摘要传 LLM 要求「更新」而非重写（_previous_summary），信息跨多次压缩保留。

最常见陷阱：摘要模型上下文窗口必须 ≥ 主模型，否则报超限、_generate_summary() 返回 None，压缩器无摘要丢弃中段，静默损失上下文。
▸可插拔 ContextEngine（源码确认）context_engine.py 
class ContextEngine(ABC):
    # run_agent.py 直接读取的 token 状态
    last_prompt_tokens=0; threshold_tokens=0
    context_length=0; compression_count=0
    threshold_percent=0.75; protect_first_n=3; protect_last_n=6
    @abstractmethod
    def update_from_response(self, usage): ...
    @abstractmethod
    def should_compress(self, prompt_tokens=None)->bool: ...
    @abstractmethod
    def compress(self, messages, current_tokens=None, focus_topic=None): ...
    # 生命周期 / 可选工具 / update_model 模型切换

解析顺序：plugins/context_engine/<name>/ → 通用插件 → 内置 ContextCompressor。插件引擎绝不自动激活，须显式配置 context.engine。

▸Anthropic Prompt 缓存（system_and_3）prompt_caching.py · −75% 输入成本

Anthropic 每请求最多 4 个 cache_control 断点，Hermes 用 system_and_3：

断点 1：系统 prompt          （全程稳定）
断点 2：倒数第 3 条非系统消息  ┐
断点 3：倒数第 2 条非系统消息  ├ 滚动窗口
断点 4：最后一条非系统消息     ┘

系统 prompt 全程缓存；中段增删使其后失效；压缩后系统 prompt 缓存存活，滚动窗口 1–2 轮重建；TTL 默认 5m，长会话用 1h。Claude 模型经原生 API 或 OpenRouter 时自动启用。

工具系统（Tool Runtime）

70+ 自注册函数，按 ~28 个 toolset 分组，经中央 registry 分发；导入时 AST 自动发现，check_fn 门控可用性。

tools/registry.py （无依赖，被所有工具导入） ▲ tools/*.py （各自导入时调 registry.register()） ▲ model_tools.py （导入 registry + 触发 AST 工具发现） ▲ run_agent.py / cli.py / batch_runner.py / environments/
图 3 · 导入时注册的文件依赖链

🔎 注册与发现

每文件导入时调 registry.register(name,toolset,schema,handler,check_fn,…) → ToolEntry 存入单例。discover_builtin_tools() 用 AST 解析找含顶层 register() 的模块并导入 —— 新增工具自动发现，可选工具导入错误被捕获不影响其它。之后再发现 MCP 与插件工具。

✅ 可用性 check_fn

返回 True 才可用（API key 存在 / 服务运行 / 二进制安装）。get_definitions() 构建 schema 时运行；结果按调用缓存，抛异常视为不可用（fail-safe）。

🧱 双层错误包装

registry.dispatch() 捕获返回 {"error":"Tool execution failed…"}；handle_function_call() 二次包装。模型永远收到良构 JSON，绝不抛未处理异常。异步经 _run_async() 桥接（CLI 持久循环 / 网关一次性线程）。

🛡️ 危险命令审批

approval.py 的 DANGEROUS_PATTERNS 覆盖 rm -rf、mkfs/dd、无 WHERE 的 DROP/DELETE、>/etc/、curl|sh、fork 炸弹等。CLI 交互询问 / 网关异步回调 / 可选辅助 LLM 智能放行；按会话记录，永久允许写入 command_allowlist。

终端/运行时后端

localdockersshdaytonamodalsingularity+ vercel_sandbox

支持按任务 cwd 覆盖、后台进程管理、PTY、审批回调。Daytona / Modal 提供 serverless 持久化。

06

Provider 运行时解析

跨 CLI / Gateway / Cron / ACP / 辅助调用共享的统一解析器，把 (provider, model) 映射为 (api_mode, api_key, base_url)。

📐 解析优先级

① 显式 CLI/运行时请求 → ② config.yaml 配置 → ③ 环境变量 → ④ provider 默认/自动。

把已保存的选择视为真相来源，防止过期 shell export 静默覆盖用户最后选的端点。

🔐 凭据按 base URL 隔离

OPENROUTER_API_KEY 只发 openrouter.ai；AI_GATEWAY_API_KEY 只发 vercel.sh；OPENAI_API_KEY 用于自定义端点与兜底。避免把错误 key 泄漏给自定义端点。

07

会话存储（SQLite + FTS5）

单库 ~/.hermes/state.db（WAL 模式）持久化会话元数据、完整消息历史与模型配置，取代早期逐会话 JSONL 方案。源文件 hermes_state.py。

state.db (SQLite, WAL) ├── sessions 元数据·token·计费 ├── messages 完整消息历史 ├── messages_fts FTS5 全文 ├── messages_fts_trigram │ trigram（CJK/子串） ├── state_meta 键值元数据 └── schema_version 迁移版本 = 11

🧬 关键设计

WAL 多读单写（网关多平台并发）。

FTS5 全文 + trigram 分词支持中文/子串搜索。

会话血缘经 parent_session_id 链（压缩触发的会话分裂产生父子链），递归 CTE 上溯/下溯。

source 标签（cli/telegram/…）做平台过滤；批量与 RL 轨迹不存此库。

写竞争处理（多进程共享一库）：短超时 1 秒（非默认 30 秒）+ 应用层带随机抖动重试（20–150ms，最多 15 次）+ BEGIN IMMEDIATE 即时暴露锁竞争 + 每 50 次写做 PASSIVE WAL checkpoint。避免 SQLite 确定性退避导致的 convoy effect。

全文搜索能力

支持 FTS5 语法（隐式 AND、"短语"、OR、NOT、前缀*），自动 _sanitize_fts5_query()（剥不匹配引号、连字符词包引号、去悬空布尔）；结果含 >>>match<<< 片段与前后各 1 条上下文。

08

自我改进学习闭环 ★ 差异化核心

由四个协作子系统编织成的回路，让 agent 把一次性经验沉淀为可复用、可维护、可回收的长期资产。

图 4 · 闭合学习回路
▸技能系统 Skills（写入端）skill_manage · agentskills.io

渐进披露三级：skills_list()(~3k tok 概览) → skill_view(name)(完整) → skill_view(name,path)(参考文件)。每个技能自动成为斜杠命令。

agent 自管（闭环关键）：摸索出非平凡工作流时存为技能。触发：复杂任务(5+ 工具)后、踩坑找到可行路径、被用户纠正、发现非平凡工作流。动作 create / patch（首选）/ edit / delete / write_file / remove_file。

Skills Hub：官方 / skills.sh / well-known / GitHub tap（openai·anthropics·huggingface·NVIDIA）/ ClawHub / LobeHub / browse.sh / 直链 URL；全部安全扫描，分 builtin/official/trusted/community 四级信任，--force 不能越过 dangerous 裁定。Bundles 用 YAML 把多技能聚到一条命令，不打断缓存。

▸Curator（agent 技能的后台维护）curator.py · 永不自动删除

只管agent 创建的技能，绝不碰内置/Hub 技能，最坏只归档（可恢复）。空闲触发：距上次 > interval_hours(7天) 且空闲 > min_idle_hours(2h) 才 fork 后台 AIAgent。

两阶段：① 确定性转换（无 LLM）—— 未用满 30 天转 stale，90 天移入 .archive/；② LLM 评审（max_iterations=8）—— 逐技能保留/patch/合并/归档。

遥测 .usage.json（use/view/patch 计数 + 状态 + pinned）；每次真跑前 tar.gz 快照（默认留 5），rollback 一键回滚；pin 让技能免于转换与删除。

▸持久记忆 + MemoryManagermemory_manager.py · 内置 + 1 外部
文件用途上限
MEMORY.md环境事实·约定·习得经验2,200 字符 (~800 tok)
USER.md用户画像·偏好·沟通风格1,375 字符 (~500 tok)

memory 工具动作 add/replace/remove（子串匹配），无 read（已在 prompt）；满额报错要求先合并；自动拒重复 + 注入/外泄安全扫描。

MemoryManager（源码确认）：内置提供器恒首位不可移除，外部只允许一个（多注册即告警拒绝）。所有注入记忆包进 <memory-context> 围栏 + 系统注记；StreamingContextScrubber 跨流式 chunk 防标签拆分泄漏（宁可截断也不泄漏）。任一提供器失败不阻塞其它。

外部提供器（8 个）：Honcho、OpenViking、Mem0、Hindsight、Holographic、RetainDB、ByteRover、Supermemory —— 与内置并存而非替代。

▸Honcho dialectic 用户建模两层注入 · 三正交旋钮 · 多趟深度

在对话之后推理，维护「用户是谁」运行模型。两层注入：基础层（会话摘要+用户表征+peer card+AI 身份）+ dialectic 补充层（LLM 综合「当下最相关」）。

三正交旋钮：contextCadence(基础刷新,默1)·dialecticCadence(LLM 间隔,默2,建议1–5)·dialecticDepth(每次1–3趟：Pass0 冷/暖启动·Pass1 自审补缺·Pass2 调和矛盾，强信号提前 bail)。

会话启动后台 prewarm；按查询长度自适应推理档位（minimal→max）。五工具 honcho_profile/search/context/reasoning/conclude；观察模式 directional（全互观，启用跨 peer dialectic）/ unified（共享池）。

🔑 会话键 + 两级守卫

键格式 agent:main:{platform}:{chat_type}:{chat_id}，统一用 build_session_key() 构造。

Level 1（base 适配器）查 _active_sessions 活动则入队+置中断；Level 2（runner）查 _running_agents 拦截 /stop /new /queue /status /approve /deny，/approve 内联分发避竞态。

独占凭据适配器用 acquire/release_scoped_lock() 防两 profile 同用一 bot token。

🪝 钩子系统

事件 gateway:startup · session:start/end/reset · agent:start/step/end · command:*。来自 builtin_hooks/（常驻）与 ~/.hermes/hooks/（含 HOOK.yaml + handler.py）。

📤 投递路径

直接回复 / home channel（cron 与后台结果）/ 显式目标（telegram:-100…）/ 跨平台。Cron 投递不镜像进网关历史（避免交替违规）。

10

定时任务（Cron）

一等公民的「agent 任务」（非 shell）。cron/jobs.py + cron/scheduler.py，默认 60 秒 tick。

📅 四种调度格式

相对延迟 30m/2h/1d间隔 every 2hcron 0 9 * * *ISO 时间戳

模型侧单一 cronjob 工具：create/list/update/pause/resume/run/remove。任务存 jobs.json（写临时再 rename 原子语义），状态 scheduled/paused/completed/running。

🧱 隔离与防护

全新会话隔离：无历史、prompt 须自包含（不能反问）。

递归守卫：cron 会话禁用 cronjob toolset，防自建任务爆 token。

跨进程锁：fcntl.flock/msvcrt.locking 防重复执行同批。

技能/脚本支撑：可挂载多技能（按序注入为上下文）；可挂 Python 脚本，每轮前运行、stdout 注入 prompt 做数据采集/变更检测（超时默认 120s）。provider 恢复：run_job() 传入 fallback + 凭据池，支持回退与密钥轮换。[SILENT] 前缀抑制投递（仅副作用）。
11

ACP 编辑器集成

把同步 AIAgent 包成异步 JSON-RPC stdio 服务，将 Hermes 暴露为编辑器原生 agent（VS Code / Zed / JetBrains）。

hermes acp / hermes-acp / python -m acp_adapter → entry.main() → 解析 --version/--check/--setup → 载入 .env → 构造 HermesACPAgent → acp.run_agent(use_unstable_protocol=True) （stdout 专供 JSON-RPC，日志走 stderr）

🧩 核心组件

HermesACPAgent（协议实现）· SessionManager（线程安全跟踪 session/agent/cwd/cancel_event）· 事件桥 · 权限桥 · 工具渲染。

🌉 事件 / 权限桥

agent 在工作线程、I/O 在主循环 → run_coroutine_threadsafe。危险审批映射 ACP 权限（超时/失败默认拒绝）；事件桥按工具名 FIFO 正确归位并行/重复同名调用。

🔁 会话生命周期

new(cwd) 绑定 cwd → prompt 在线程池跑 → cancel 置事件 + interrupt → fork 深拷历史。复用 Hermes 运行时解析器，会话进共享 state.db 可被 session_search 发现。

横切设计原则与总结

📌 Prompt 稳定性

系统 prompt 中途不变；除显式 /model 外无破缓存变更。

👁️ 可观测执行

每个工具调用经回调对用户可见（CLI spinner / 网关消息）。

⏸️ 可中断

API 与工具执行可被输入或信号中途取消。

🧩 平台无关内核

一个 AIAgent 服务全部入口，平台差异只在入口层。

🔗 松耦合

MCP/插件/记忆/RL 用 registry + check_fn 门控，而非硬依赖。

🪪 Profile 隔离

每 profile 独立 HERMES_HOME/配置/记忆/会话/PID，可并发。

总评

Hermes 的架构可概括为「一个稳定可缓存的内核 + 一圈松耦合的可选子系统 + 一条贯穿其间的自我改进回路」：

  • 内核（AIAgent）把 prompt 组装/provider 解析/工具分发/压缩缓存/持久化收敛为单一同步引擎，以回调面适配多入口——这是「一套代码服务 CLI/网关/编辑器/训练」的根因。
  • 可缓存优先是反复主题：分层冻结、记忆快照、system_and_3 断点、双层压缩阈值差异化，处处为「稳定 prefix → 高命中 → 低成本」服务。
  • 学习闭环（Skills + Curator + Memory + 可选 Honcho）让经验沉淀为长期资产，是相对传统无状态框架的本质差异。
  • 运行时可移植性（6 后端 + serverless + ~20 平台 + ACP + Cron）与研究闭环（Atropos + 基准 + 轨迹 + Tinker）分别覆盖「随处部署」与「自我训练下一代模型」。

最值得借鉴：导入时 AST 自动发现 + check_fn 门控的工具系统、双层错误包装保证良构 JSON、可中断后台线程 API 调用、<memory-context> 流式围栏防注入泄漏、SQLite 抖动重试规避 convoy effect、把上下文压缩抽象为可插拔 ContextEngine 的工程克制。
